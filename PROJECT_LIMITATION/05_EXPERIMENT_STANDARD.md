# A-Share Quant V1 Experiment Standard

## 目的

本文件定义一次“有效实验”必须满足的记录、比较和复现要求。

它的目标是防止以下问题：

1. 改了代码但不知道到底改了什么
2. 结果变好却不知道原因
3. 不同实验之间不可比较
4. 凭印象保留或淘汰策略

---

## 核心原则

1. 没有记录的实验，等同于没有做
2. 没有统一基线的对比，等同于不可比较
3. 没有可复现配置的结果，等同于不可采信
4. 没有解释来源的收益改善，默认不晋级

---

## 实验类型

### 1. Baseline Run

用途：

1. 建立当前协议版本下的标准对照
2. 为后续改动提供固定参照

### 2. Candidate Run

用途：

1. 比较新规则、新参数、新打分逻辑
2. 在不污染基线的前提下做探索

### 3. Validation Run

用途：

1. 做样本外验证
2. 做滚动或 walk-forward 验证
3. 验证结果稳健性

### 4. Regression Run

用途：

1. 验证重构、修 bug、成本模型变更是否破坏历史行为

---

## 每次实验必须记录的内容

每个实验必须具备唯一 `run_id`，并至少记录：

1. `run_id`
2. `run_type`
3. `run_timestamp`
4. `operator`
5. `code_version`
6. `protocol_version`
7. `config_paths`
8. `data_source`
9. `data_range`
10. `sample_segmentation_method`
11. `strategy_family`
12. `strategy_variant`
13. `key_parameters`
14. `cost_model_version`
15. `limit_model_version`
16. `metrics_summary`
17. `report_path`
18. `notes`

建议保存为结构化文件：

1. `yaml`
2. `json`
3. `csv` 汇总索引

---

## 每次重要改动必须附带的研究说明

凡是影响研究结论的改动，必须在 [08_RESEARCH_JOURNAL.md](./08_RESEARCH_JOURNAL.md) 留痕，并至少说明：

1. 改了什么
2. 为什么改
3. 预期改善什么
4. 实际改善了什么
5. 结果是否有副作用
6. 下一步如何验证

---

## 必须重跑基线的改动

出现以下情况之一，必须重跑 baseline：

1. 样本切分逻辑变化
2. 主线定义或板块打分逻辑变化
3. 个股分层逻辑变化
4. 趋势过滤、入场、持有、退出逻辑变化
5. 成本、滑点、涨跌停或成交假设变化
6. 数据清洗、复权、停牌、新股、ST 处理变化
7. 指标公式变化
8. 回测引擎核心行为变化

---

## 不允许的实验行为

1. 看到结果不好，临时改规则重新跑但不记录
2. 只展示最好的一次结果
3. 修改数据口径后沿用旧结论
4. 在同一结论中混用不同协议版本结果
5. 把一次性幸运结果当成长期有效证据

---

## 合法比较的条件

只有在以下条件基本一致时，实验结果才可直接比较：

1. 数据范围一致
2. 样本切分逻辑一致
3. 数据处理口径一致
4. 成本与成交假设一致
5. 指标定义一致
6. 报告格式一致

若不一致，必须明确标注“不可直接比较”。

---

## 实验产物落点

建议固定以下目录：

1. `experiments/runs/` 保存结构化 run 记录
2. `reports/` 保存回测报告
3. `PROJECT_LIMITATION/07_DECISION_LOG.md` 记录关键决策
4. `PROJECT_LIMITATION/08_RESEARCH_JOURNAL.md` 记录研究结论

---

## 最小通过标准

一次实验若要被认为“有效”，至少需要：

1. 有唯一 `run_id`
2. 有完整配置快照
3. 有核心指标结果
4. 能从记录追溯到代码版本与报告
5. 能说明该实验相对 baseline 的变化点

---

## 一句话原则

实验不是“跑出一个结果”，而是“产出一个可比较、可追溯、可复现的证据单元”。
